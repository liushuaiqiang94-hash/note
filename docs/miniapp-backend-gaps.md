# Miniapp 后端能力缺口说明

## 1. 说明

本文档用于区分：

- 已在 `apps/server` 中实现、可以直接联调的能力
- 需求文档里有，但当前后端还没有实现的能力

miniapp 开发应优先依据当前已实现接口，不要按需求文档假设不存在的接口。

## 2. 已实现能力

当前后端已经可联调的能力：

- 微信登录换 JWT
- 获取当前用户资料
- 分类列表、新增、修改、删除
- 任务列表、新增、详情、修改、删除
- 任务状态切换
- 今日任务列表
- 回收站列表与恢复
- 统计概览与趋势
- 基于服务内定时任务的过期扫描、回收站清理、提醒扫描位点

## 3. 未实现或只实现一部分的能力

| 需求能力 | 当前状态 | miniapp 处理建议 |
| --- | --- | --- |
| 微信订阅消息提醒 | 未真正接入消息发送，只实现了提醒记录与扫描位点 | miniapp 先不要依赖服务端“消息已发送”能力，提醒开关页面可暂缓或标注开发中 |
| 重复任务自动生成 | 仅有 `repeatType` 字段，未实现周期实例生成 | 表单可以先保留字段，但不要在页面承诺周期自动创建效果 |
| 分类占比统计 | 当前只有概览和趋势接口，没有分类占比接口 | 统计页先做完成数、完成率、趋势，不做分类占比 |
| 日历视图专用接口 | 没有单独的日历接口 | 日历页可基于 `GET /tasks?date=yyyy-MM-dd` 逐日查询，或等后端补聚合接口 |
| 提醒设置页能力 | 没有提醒模板、订阅状态、提醒开关接口 | “提醒设置”页先不开发，或只做静态占位 |
| 反馈接口 | 未实现 | “意见反馈”入口先隐藏或只留本地占位 |
| 关于我们接口 | 未实现 | “关于我们”使用前端静态内容 |
| 管理端接口 | 未实现 | miniapp 不依赖后台能力 |

## 4. 与需求文档的主要差异

### 4.1 统计能力差异

需求文档中提到：

- 完成趋势
- 分类占比
- 完成率统计

当前后端只实现：

- `GET /stats/overview`
- `GET /stats/trend`

未实现：

- 分类占比

### 4.2 提醒能力差异

需求文档中提到：

- 截止前提醒
- 订阅消息提醒

当前后端只实现：

- 任务保存时记录提醒时间
- 定时扫描待提醒记录

未实现：

- 真正向微信发送订阅消息

### 4.3 重复任务差异

需求文档中提到：

- 每天、每周、每月重复
- 到新周期后自动生成新任务或重置状态

当前后端只实现：

- `repeatType` 字段

未实现：

- 周期实例生成
- 周期任务重置

### 4.4 我的页面差异

需求文档中提到：

- 用户信息
- 提醒设置
- 意见反馈
- 关于我们

当前后端只实现：

- 用户信息接口 `/me`

未实现：

- 提醒设置接口
- 反馈接口
- 关于我们接口

## 5. miniapp 开发建议

- 第一阶段只接当前可用能力：登录、任务、分类、统计
- 对“提醒设置”“反馈”“关于我们”“高级统计”做前端占位，不依赖后端接口
- 如果页面里要展示“重复任务”，先仅做字段展示，不承诺自动生成能力
- 统计页产品文案不要写“分类占比”，除非后端后续补接口
